# DataLad sub-dataset integration for codio code library

## Goal

Add DataLad sub-dataset support to codio so that the code library directory (code/lib/) is scaffolded and managed as a DataLad dataset, with CLI commands to initialize, add, and update library repos as sub-datasets.

## Prompts

1. Call `project_context()` and `runtime_conventions()` to understand the current workspace, then read `src/codio/scaffold.py` and `src/codio/cli.py` to understand the existing scaffold and CLI structure. Also read `src/codio/config.py` and `src/codio/models.py` to see how paths and catalog entries are modeled.
2. Implement DataLad sub-dataset support: (1) In `src/codio/scaffold.py`, update `init_codio_scaffold` to run `datalad create` on the code library root directory (e.g. `code/lib/`) if DataLad is available, making it a sub-dataset of the parent repo. (2) In `src/codio/cli.py`, add a `codio lib add` subcommand that takes a repo URL and name, runs `datalad install` (or `datalad clone`) to add it as a sub-dataset under `code/lib/<name>/`, and registers the entry in `.codio/catalog.yml`. (3) Add a `codio lib update` subcommand that runs `datalad update` across all sub-datasets in `code/lib/`. Use subprocess calls guarded by a `shutil.which('datalad')` check so codio degrades gracefully when DataLad is absent.
3. Add tests in `tests/` for the new DataLad scaffold path and `lib add`/`lib update` commands (mock the `datalad` subprocess calls). Run `make test` to verify all tests pass.
4. Call `note_create(note_type='task')` to record this implementation as a completed task, documenting: the new `codio lib add` and `codio lib update` commands, the graceful degradation behavior when DataLad is absent, and any open questions about sub-dataset path conventions.
